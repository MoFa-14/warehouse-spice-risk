# Dashboard Layer 4 (Presentation Only)

This package implements the Flask dashboard and presentation layer for Warehouse Spice Risk.

Scope limits for this phase:

- Reads live telemetry from SQLite, with processed CSVs used when available
- Uses rule-based alerts only
- Displays stored 30-minute forecasting outputs generated by the gateway forecasting CLI
- Does not train models, generate forecasts itself, or run heavy ML workloads inside Flask

## Expected Data Layout

The dashboard reads data from the repository `data/` folder:

- Primary live source: `data/db/telemetry.sqlite`
- `data/raw/pods/<pod_id>/YYYY-MM-DD.csv`
- `data/raw/link_quality/YYYY-MM-DD.csv` (optional)
- `data/processed/pods/<pod_id>/YYYY-MM-DD_processed.csv` (optional)

The dashboard now prefers live SQLite data for raw telemetry and link quality. Processed CSV files are still used when available for cleaned historical views. If the SQLite database is missing, the dashboard falls back to the raw/link CSV files.

## Install

From the repository root:

```powershell
.\.venv\Scripts\python.exe -m pip install -r .\dashboard\requirements.txt
```

## Run

From the repository root:

```powershell
.\scripts\run_dashboard.ps1
```

Or run it manually from the dashboard directory:

```powershell
cd .\dashboard
..\.venv\Scripts\python.exe -m flask --app app.main run --debug
```

Open `http://127.0.0.1:5000` in your browser.

Once this SQLite-aware code is running, you do not need to restart Flask for each new sensor reading. The dashboard queries the database on each request, but timed page reload is now disabled by default so the UI does not jump every few seconds. You only need to restart the dashboard once after changing the dashboard code itself.

If you want timed page reload for a kiosk-style display, start the dashboard with an explicit refresh interval:

```powershell
.\scripts\run_dashboard.ps1 -AutoRefreshSeconds 5
```

Or set the environment variable before launching Flask manually:

```powershell
$env:DASHBOARD_AUTO_REFRESH_SECONDS = "5"
..\.venv\Scripts\python.exe -m flask --app app.main run --debug
```

## Time Display

Telemetry and forecast timestamps remain stored in UTC in SQLite and CSV/JSONL outputs.

The dashboard now displays those timestamps in the configured local display timezone instead of forcing UTC in the UI. By default it uses the local system timezone, which avoids the one-hour BST offset problem on UK daylight-saving days.

The default config lives in `dashboard/app/config.py`:

- `DISPLAY_TIMEZONE = "local"`

This affects:

- latest-reading timestamps
- alert timestamps
- health page timestamps
- forecast timestamps
- chart x-axis labels
- custom date-range form labels and parsing

## Pages

- `/` overview with latest readings per pod
- `/pods/<pod_id>` pod detail with historical charts
- `/health` link quality and data quality summary
- `/alerts` active and acknowledged rule-based alerts
- `/prediction` latest stored baseline and event-persist forecasts for every pod with saved forecast rows

## Forecast Display

The dashboard reads stored forecasting results from the same live SQLite database when available:

- `forecasts`
- `evaluations`

If those tables are not present, it falls back to:

- `data/ml/forecasts.jsonl`
- `data/ml/evaluations.jsonl`

The dashboard does not run forecasting itself. It only renders the most recent stored output per pod, including:

- baseline forecast scenario
- event-persist scenario when a disturbance was detected
- rule-based predicted alert levels such as `OPTIMAL`, `GOOD/ACCEPTABLE`, `WARNING`, `HIGH RISK`, and `CRITICAL`
- the same recommendation text used by the live-reading cards, now applied to the predicted 30-minute trajectory
- event metadata and reason text
- latest evaluation metrics when the 30-minute horizon has already completed

The `/prediction` page now shows:

- a summary card for each pod with stored forecast output
- a full forecast panel for each pod on the same page
- a worst-case predicted alert badge and action message for each forecast scenario
- quick jump links so you can move directly to pod `01`, pod `02`, and so on without opening each pod detail page separately

To populate the Prediction page, run the forecasting CLI first from the repository root:

```powershell
python -m gateway.cli.forecast_cli once --pod 01
python -m gateway.cli.forecast_cli run --all --every-minutes 30
```

## Alert Levels

The dashboard uses these rule-based alert levels:

- Level 0 `OPTIMAL`: temperature and humidity within ideal storage range
- Level 1 `GOOD/ACCEPTABLE`: safe but slightly outside optimal
- Level 2 `WARNING`: likely to harm quality if sustained
- Level 3 `HIGH RISK`: strong risk of caking and quality loss
- Level 4 `CRITICAL`: rapid spoilage or mold risk

Threshold constants implemented in `dashboard/app/services/thresholds.py`:

- Temperature: optimal `10-21 C`, warning above `22 C`, critical at `25 C`
- Relative humidity: low warning below `30%`, ideal `30-50%`, warning above `50%`, high risk above `60%`, critical at `65%`

If both temperature and humidity rules apply, the dashboard takes the maximum severity.

## Acknowledgements

Alert acknowledgements are stored locally in:

- `dashboard/app/runtime/acks.json`

This is a lightweight UI-only mute for 30 minutes by default. It does not affect CSV logging.

## Notes

- Alerts are rule-based only in this phase.
- Dew point is displayed from live SQLite raw readings by computing it from temperature and humidity, with processed CSVs still used as an optional fallback.
- If the newest sample is incomplete, the dashboard shows the latest sample timestamp and also shows the last complete measurement timestamp for context.
- Dew point does not generate alerts yet.
- Prediction charts are display-only and reflect the latest stored gateway forecast output.
